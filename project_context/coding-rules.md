# Coding Rules

## 문서 목적
이 문서는 LLM 구현자가 approval-factory MVP의 코드를 작성할 때 따라야 할 기본 규칙을 정의한다.

## 최상위 원칙
- 작은 함수
- 작은 모듈
- 명시적 계약
- 테스트 동반
- 문서와 코드 정합성 유지

## 파일/모듈 규칙
- 한 파일은 한 책임에 집중한다.
- 한 모듈이 여러 역할을 동시에 가지지 않도록 한다.
- 설계 문서에서 정의한 경계를 넘는 import를 피한다.

## 함수 규칙
- 함수는 가능한 짧고 명확하게 유지한다.
- 숨은 상태 변경보다 명시적 입력/출력을 선호한다.
- 예외가 발생할 수 있으면 의미 있는 메시지를 남긴다.

## 타입 규칙
- 공개 함수와 핵심 도메인 모델에는 타입 힌트를 사용한다.
- WorkItem, PRPlan, ApprovalRequest, EvidenceBundle 같은 핵심 모델은 명시적 구조를 가진다.

## 테스트 규칙
- 구현 변경에는 가능한 한 테스트를 동반한다.
- 상태 전이, approval package 생성, docs sync 판정 같은 핵심 로직에는 단위 테스트를 우선 작성한다.
- smoke 수준의 end-to-end 흐름은 별도 테스트 또는 샘플 run으로 검증한다.

## 로그/출력 규칙
- 출력은 사람이 읽을 수 있어야 한다.
- approval package와 run artifacts는 구조화된 파일로 남겨야 한다.
- 디버그 목적의 임시 출력은 남기지 않는다.

## 금지 사항
- 승인 없는 설계 변경
- 다른 PR 범위 침범
- 테스트 없는 핵심 로직 추가
- 문서 계약 무시
- 과도한 추상화

## 변경 시 문서 체크
다음 변경은 문서 영향도를 반드시 확인한다.
- 도메인 모델 변경
- 상태 전이 변경
- 승인 게이트 변경
- evidence 형식 변경
- docs sync 정책 변경
